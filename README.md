# Smart Grid Forecasting

Проект реализует полный исследовательский пайплайн для прогнозирования почасового электропотребления городской инфраструктуры. Ключевые направления работы: сравнение нейросетевых архитектур (LSTM, Transformer, PatchTST, TFT) с базовыми методами (Ridge, XGBoost), анализ остатков прогноза и экономическая оптимизация накопителя энергии (BESS).

## 1. Задача и постановка проблемы

### 1.1 Формальная постановка

Пусть `y_t` — почасовое потребление электроэнергии в момент `t`. Задача multi-step forecasting:

```
ŷ_{t+1}, ..., ŷ_{t+H} = f(y_{t-L+1}, ..., y_t, x_{t-L+1}, ..., x_t)
```

где `H = 24` (горизонт прогноза, часов), `L = 192` (длина истории, часов), `x_t` — вектор из 26 ковариат (погода, тарифные зоны, EV-нагрузка и т.д.).

### 1.2 Практическая мотивация

Для диспетчера энергосистемы точный суточный прогноз позволяет:
- снизить резервирование генерирующих мощностей;
- оптимизировать зарядку/разрядку батарейного накопителя;
- сократить потребление в пиковые тарифные зоны (demand charge);
- предотвращать перегрузки сети при экстремальных температурах.

Ошибка прогноза на 1% от среднего потребления (~570 кВт·ч при 57 МВт·ч/ч) в масштабе месяца составляет ~400 МВт·ч — стоимостной эквивалент нескольких миллионов рублей.

### 1.3 Почему задача нетривиальна для нейросетей

Ряд потребления обладает несколькими доминирующими паттернами: суточной периодичностью (ACF(24)≈0.84), недельной (ACF(168)≈0.79), сезонным трендом (зима/лето разрыв ~59%). Классические методы (Ridge regression с lag-признаками) хорошо справляются с линейными зависимостями. Нейросети получают преимущество за счёт моделирования нелинейных взаимодействий: EV-нагрузка×температура, DSR×день_недели, промышленный спрос×смены заводов.

## 2. Синтетическая генерация данных

### 2.1 Физическая модель потребления

Потребление на каждом часовом шаге `t` строится по формуле:

```
C(t) = B(t) × W(t) × D(t) × S(t) × K(t) × G(t) × A(t) × (1 + AR(t))
       + EV(t) + Industry(t) − Solar(t) + B(t) × DSR_rebound(t)
       − C(t) × DSR_strength(t)
```

| Компонент | Обозначение | Описание |
|---|---|---|
| Базовый профиль | `B(t)` | Агрегат домохозяйств трёх типов (ранние/стандарт/ночные) |
| Погода | `W(t)` | Квадратичная реакция на температуру + влажность + ветер |
| Районы | `D(t)` | Взвешенная сумма 12 городских районов с разными профилями |
| Сезонность | `S(t)` | Косинусный дрейф; зима +15%, лето −10% |
| Праздники | `K(t)` | Снижение до −35% в новогодние каникулы |
| Стресс сети | `G(t)` | Нелинейный latent state (температура × EV × пиковые часы) |
| Аномалии | `A(t)` | Случайные всплески (+20–40%) и провалы (×0.15) |
| AR-шум | `AR(t)` | GARCH-процесс φ=0.65, σ зависит от часа и температуры |

### 2.2 Компоненты нагрузки

**Электромобили (EV).** Каждая из 2500×0.50=1250 единиц заряжается по вероятностной модели: ночные сессии (22:00–06:00, мощность 11–22 кВт, Level 2), коммерческий парк (3–5 грузовиков, 30–50 кВт/ч). При T < −15°C вероятность зарядки растёт из-за range anxiety. Исправленный алгоритм (v6) корректно переносит ночные сессии через полночь без time-wrap бага.

**Солнечная генерация.** Профиль: гауссов пик в 12:30, амплитуда зависит от сезона (sin) и облачности (1 − 0.85 × cloud^0.7). Суммарная мощность: до 1874 кВт при 0.22 × 2500 = 550 панелях.

**Промышленность.** 8 заводов моделируются полу-марковским процессом со сменным расписанием. Мощность 500–2500 кВт, вероятность работы в будний день 0.65–0.90.

**DSR (Demand Side Response).** 15 событий/год, длительность 4–8 часов, снижение 20–40%. После снятия ограничения моделируется rebound-эффект (экспоненциальный рост на 2–4 часа, амплитуда 8–16%).

### 2.3 Валидация данных

После генерации выполняются автоматические проверки:
- Отсутствие NaN/Inf и отрицательных значений.
- Утренний пик в 6:00–11:00, вечерний в 17:00–22:00.
- Соотношение выходные/будни ≈ 0.90.
- ACF(24) ≥ 0.35, ACF(168) ≥ 0.15 (при длине > 2 недель).
- Зима > Лето по среднему потреблению.

### 3.1 Вектор признаков (N_FEATURES = 26)

| Индекс | Признак | Описание |
|---|---|---|
| 0 | `consumption` | Нормализованное потребление (MinMaxScaler) |
| 1–2 | `hour_sin`, `hour_cos` | Циклическое кодирование часа суток |
| 3–4 | `is_peak`, `is_night` | Тарифные зоны |
| 5–6 | `is_weekend`, `is_holiday` | Тип дня |
| 7–8 | `temperature`, `temperature_sq` | Погодные признаки |
| 9 | `tariff_enc` | Тариф (0.0/0.5/1.0 для ночь/день/пик) |
| 10 | `doy_norm` | День года / 364 (сезонный тренд) |
| 11–12 | `humidity`, `wind_speed` | Метеопараметры |
| 13–14 | `rolling_mean_24h`, `rolling_std_24h` | Скользящие статистики |
| 15–17 | `lag_24h`, `lag_48h`, `lag_168h` | Лаговые признаки потребления |
| 18–19 | `dow_sin`, `dow_cos` | Циклический день недели |
| 20–21 | `month_sin`, `month_cos` | Циклический день года |
| 22 | `cloud_cover` | Облачность |
| 23–24 | `ev_load_norm`, `solar_gen_norm` | EV-нагрузка и солнечная генерация |
| 25 | `dsr_active` | Флаг события DSR |

### 3.2 Seasonal Naive (Dual)

При `seasonal_diff=True` целевая переменная заменяется отклонением от взвешенного seasonal naive:

```
Y_naive(t) = 0.60 × Y(t−24) + 0.40 × Y(t−168)
Y_diff(t)  = Y(t) − Y_naive(t)
```

Веса обоснованы ACF: `w_24 = ACF(24) / (ACF(24) + ACF(168)) = 0.84 / 1.63 ≈ 0.52` (округлено до 0.60). Активируется при `history_length ≥ 168`. В optimal mode (`history=192`) оба лага всегда доступны.

При `seasonal_diff=False` (рекомендуется в optimal mode) нейросети используют Seasonal Skip — явный обучаемый mix с lag-24h и lag-168h внутри архитектуры.

### 3.3 Разбивка на выборки

Хронологическое разделение без перемешивания: Train 70% / Val 15% / Test 15%.  
Все scaler'ы обучаются исключительно на тренировочной выборке (нет утечки данных).

## 4. Архитектуры моделей

### 4.1 TCN-BiLSTM-Attention (LSTM v12)

Гибридная архитектура, объединяющая локальное и глобальное моделирование временного ряда.

```
Input (192, 26)
  │
  ├─ RevIN: нормализация канала 0 (consumption) по окну
  ├─ LayerNorm("input_ln")
  │
  ├─ TCN: 4 параллельные ветки с dilation=[1,2,4,8]
  │       Conv1D(64, k=3, causal) × 2 + LayerNorm + GELU + residual
  │       Рецептивное поле: 2^4 × (k−1) = 240 часов
  │
  ├─ BiLSTM(128, return_seq=True) → LayerNorm
  │   Объединяет прямой и обратный ход: dim = 256
  │
  ├─ TemporalAttention(heads=4, key_dim=64)
  │   query = last token, key/value = вся последовательность
  │   Выход: context (256,)
  │
  ├─ Concat([last_bilstm, context]) → Dense(256, GELU) → Dense(128, GELU) → Dense(24)
  │
  ├─ AR-shortcut: Dense(24, no bias) на признаки последнего шага + lag(24h,48h,168h)
  │
  └─ Seasonal Skip (v12):
       final = sigmoid(logit_24) × neural + (1−sigmoid(logit_24)) × lag_24h
       final = sigmoid(logit_168) × final + (1−sigmoid(logit_168)) × lag_168h
       Инициализация: logit_24 → blend=0.60, logit_168 → blend=0.80
```

**Ключевые свойства:**
- Seasonal Skip создаёт inductive bias: модель стартует близко к seasonal naive, постепенно обучаясь на нелинейных отклонениях.
- CosineDecay LR schedule с `alpha=0.05`: стабильная сходимость за 300 эпох.
- Huber loss (δ=0.05): устойчивость к аномалиям с эксцессом Куртозис > 5.

### 4.2 Vanilla Transformer (Pre-LN, v4)

Encoder-only Transformer с Pre-Layer Normalization:

```
Input (192, 26) → Dense(128) + Conv1D(128, k=3, causal) → SinusoidalPE → LayerNorm → Dropout
  │
  ├─ 3 × PreLNEncoderBlock:
  │       LN → MHA(4 heads, key_dim=32) + StochasticDepth → residual
  │       LN → FFN(256, GELU) + StochasticDepth → residual
  │
  ├─ LearnedQueryPooling → Concat([last, context]) → Dense(128, GELU, residual) → Dense(24)
  ├─ AR-shortcut
  └─ Seasonal Residual (при use_seasonal_residual=True):
       output = σ(logit_24) × neural + (1−σ) × lag_24h
       + σ(logit_168) × (предыдущее) + (1−σ) × lag_168h  [при history≥192]
```

**Теоретическое преимущество перед LSTM:** путь градиента между любыми двумя временны́ми шагами равен O(1) операций (в LSTM — O(L)), что теоретически упрощает обучение на длинных контекстах (L=192).

### 4.3 PatchTST (v5, Nie et al., 2023)

Токенизация патчами: вместо 192 скалярных токенов — 15 суточных блоков.

```
patch_len=24, stride=12 → n_patches=(192−24)/12+1 = 15 токенов
```

RevIN применяется только к каналу consumption (канал 0). Бинарные каналы (is_weekend, is_holiday и др.) не нормализуются, так как при константном окне (std=0) деление приводит к взрыву активаций.

Каждый патч = 24-часовой профиль → Transformer учится на суточных паттернах, а не на отдельных часах.

### 4.4 Temporal Fusion Transformer Lite (Lim et al., 2021)

TFT-Lite принимает два входа:
- `series` (N, 192, 1): нормализованное потребление;
- `covariates` (N, 192, 25): все 25 дополнительных признаков.

Gated Residual Network (GRN) для каждого входа:
```
GRN(x) = LayerNorm(proj(x) + σ(W₂·ELU(W₁·x)) ⊙ W₂·ELU(W₁·x))
```
Механизм гейтинга адаптивно «отключает» нерелевантные признаки (например, EV в жаркий летний день).

### 4.5 Baseline-модели

**Ridge Regression.** Обучается на развёрнутом тензоре `(N, 192×26=4992)`. Эффективно находит линейные комбинации lag-признаков. Регуляризация L2 (α=1.0) предотвращает переобучение.

**XGBoost (Multi-Horizon).** 24 независимых дерева (одно на каждый шаг горизонта). Признаки: 192 лага consumption + rolling mean/std по всем 26 каналам (окна 6/12/24/48h) + ковариаты последнего шага. Итого ≈362 признака на модель. Early stopping по валидации.

## 5. Обучение и регуляризация

### 5.1 Loss function

Huber loss с δ=0.05 для нейросетей:

```
L_δ(r) = { r²/2,      если |r| ≤ δ
           { δ(|r|−δ/2), если |r| > δ
```

Выбор δ=0.05 обоснован высоким эксцессом Куртозис≈7–9 в распределении остатков — тяжёлые хвосты требуют робастной функции потерь.

### 5.2 Оптимизатор и расписание LR

- Adam (clipnorm=1.0) — ограничение нормы градиента предотвращает взрывной рост при длинных последовательностях.
- CosineDecay (в optimal mode): LR снижается от 8×10⁻⁵ до 4×10⁻⁶ за 300 × (12000/16) ≈ 225000 шагов.
- Fallback ReduceLROnPlateau (factor=0.5, patience=12) — при переходе в plateau.

### 5.3 Ранняя остановка

EarlyStopping по `val_loss`, patience=35, min_delta=1e-5, restore_best_weights=True. Диагностика режима обучения:
- gap < 0.02 → balanced ✓
- gap ≥ 0.02 и train_mae < 0.08 → overfitting
- train_mae ≥ 0.08 → underfitting

### 5.4 Защита от OOM

При нехватке GPU/RAM trainer автоматически уменьшает `batch_size` вдвое и перезапускает обучение. Минимальный batch_size = 4.

## 6. Оценка качества

### 6.1 Метрики прогноза

| Метрика | Формула | Интерпретация |
|---|---|---|
| MAE | `mean(|y−ŷ|)` | Абсолютная ошибка в кВт·ч |
| RMSE | `sqrt(mean((y−ŷ)²))` | Штраф за крупные отклонения |
| MAPE | `100% × mean(|y−ŷ|/|y|)` | Относительная ошибка |
| sMAPE | `200% × mean(|y−ŷ|/(|y|+|ŷ|))` | Симметричная MAPE |
| WAPE | `100% × sum(|y−ŷ|)/sum(|y|)` | Взвешенная APE |
| R² | `1 − SS_res/SS_tot` | Доля объяснённой дисперсии |
| MBE | `mean(ŷ−y)` | Систематическое смещение |

Составная оценка: `Composite = 0.5×MAE + 0.3×RMSE + 0.2×|ACF(24)|×MAE`

ACF(24) остатков штрафует модели, не устранившие суточный паттерн.

### 6.2 Диагностика остатков

Статистические тесты по тест-выборке:

| Тест | Гипотеза H₀ | Ожидаемый результат |
|---|---|---|
| ADF (Дики-Фуллер) | Ряд нестационарен | p < 0.05 (стационарны) |
| KPSS | Ряд стационарен | p > 0.05 (не отвергаем) |
| Ljung-Box (lag=10,20) | Нет автокорреляции | p > 0.05 (в идеале) |
| Durbin-Watson | Нет автокорреляции 1-го порядка | DW ≈ 2.0 |
| Jarque-Bera / Shapiro-Wilk | Нормальность | Ожидается отклонение (эксцесс) |

### 6.3 Бэктестинг

Скользящий бэктестинг на 8 непересекающихся окнах тестовой выборки. Оценивает стабильность модели во времени (нет ли деградации качества к концу периода).

### 6.4 MC Dropout (неопределённость)

30 forward pass с `training=True` дают выборку предсказаний. Стандартное отклонение пересчитывается в кВт·ч через масштаб MinMaxScaler.

## 7. Оптимизация накопителя энергии

### 7.1 Модель BESS

Батарея ёмкостью 4500 кВт·ч с мощностью зарядки/разрядки 2250 кВт. Стратегия: заряд ночью (тариф 1.80 руб/кВт·ч), разряд в пик (6.50 руб/кВт·ч).

Ограничения SOC: `SOC_min = 25%` и `SOC_max = 75%` (рабочий диапазон предотвращает ускоренную деградацию).

### 7.2 Экономика

```
Gross Savings = (Baseline Cost − Optimized Cost) + Demand Charge Savings
Net Savings   = Gross Savings − Degradation Cost − O&M Cost
Payback Years = Battery Cost / (Net Savings × 8760 / Horizon Hours)
```

Demand Charge вычисляется только по пиковым биллинговым часам (10:00–17:00 и 21:00–23:00 в будни), а не по всему суточному максимуму — это соответствует реальной тарифной структуре большинства регионов России.

**Примечание:** расчёт выполняется на первых 720 часах тест-периода. При повышенном потреблении в тест-периоде (зимний срез) срок окупаемости будет ниже среднегодового; для реалистичной оценки используйте годовой прогноз.

### 7.3 Три стратегии

| Стратегия | SOC диапазон | ΔE, кВт·ч | Износ | Рекомендуется |
|---|---|---|---|---|
| Консервативная | 40%→60% | 900 | Минимальный | Новая батарея |
| Умеренная | 25%→75% | 2250 | Средний | Оптимальный баланс |
| Агрессивная | 8%→92% | 3780 | Высокий | Максимизация дохода |
